## 결정 프로세스

### 의사결정 방법
1. **프로토타입 테스트**: 실제 구현해보고 판단
2. **데이터 기반 결정**: 플레이테스트 데이터 수집 후 결정
3. **팀 토론**: 세찬(프로그래머) + 민영(아티스트) 합의
4. **커뮤니티 피드백**: 베타 테스터 의견 반영

### 결정 기록 템플릿
```markdown
## 결정 사항: [제목]
- **날짜**: YYYY-MM-DD
- **결정 내용**: 구체적인 결정 사항
- **근거**: 왜 이렇게 결정했는지
- **대안**: 고려했던 다른 옵션들
- **영향**: 이 결정이 미치는 영향
- **재검토 일정**: 언제 다시 검토할지
```

## 결정 대기 중인 이슈들

### Issue #1: RSP 크리처 설계
**상황**: RSP 크리처의 속성과 메커니즘을 결정해야 함
**옵션**: 
- A) 인간형 (화염 트랩으로 제거)
- B) 기계형 (물 트랩으로 제거) 
- C) 노말형 (물리 트랩으로 제거)
**고려사항**: 렘페이지가 이미 기계형이므로 다양성 필요
**기한**: 2주 내 결정 필요 (민영 모델링 일정)

### Issue #2: 전력 소모 밸런싱
**상황**: 컴퓨터, 조명, CCTV, 문 닫기의 전력 소모량 결정
**현재 상태**: 모두 "테스트 필요" 상태
**테스트 계획**: 
1. 가상의 수치로 프로토타입 제작
2. 10회 이상 플레이테스트
3. 생존율 50-70% 달성하는 수치 찾기
**기한**: 프로토타입 완성 후 즉시

### Issue #3: 맵 확장 알고리즘
**상황**: 사이클별 맵 크기 증가 패턴 세부 결정
**현재**: 기본 아이디어만 있음 (반지름 25m씩 증가)
**세부 결정 필요**:
- 방 생성 밀도
- 복도 연결 복잡도
- 특수 방 출현 빈도
- 아이템 배치 알고리즘
**기한**: Phase 1 완료 후

### Issue #4: 협동 플레이 vs 솔로 플레이
**상황**: 두 모드 간 밸런싱 차이 결정
**고려사항**:
- 솔로는 더 쉬워야 하나, 더 어려워야 하나?
- 자원 배분을 어떻게 조정할 것인가?
- 크리처 수를 조정할 것인가, 능력치를 조정할 것인가?
**영향**: 전체 게임 밸런싱에 큰 영향
**기한**: Phase 2 완료 전
